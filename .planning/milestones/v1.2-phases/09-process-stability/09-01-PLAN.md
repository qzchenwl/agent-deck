  <action>
Investigate the exit 137 problem by tracing what happens when a message is sent to a Claude Code session that has a running Bash tool child process. The investigation must answer these questions:

**1. Reproduction (what happens):**
- Create a tmux session running a long-lived process (e.g., `sleep 300` or a shell script that runs for minutes)
- While it's running, use `tmux send-keys -l "some text" && tmux send-keys Enter` (the same mechanism agent-deck uses)
- Observe: Does the long-running process get killed? What signal does it receive?
- Try this both with a raw shell (no Claude Code) and describe what would differ with Claude Code's PTY handling

**2. Signal chain analysis (who sends SIGKILL):**
- Trace the full path: agent-deck `handleSessionSend` -> `sendWithRetry` -> tmux `SendKeysAndEnter` -> tmux server -> terminal PTY -> Claude Code's Node.js process -> Bash tool child
- Identify at which layer the SIGKILL originates. Key question: Does tmux send-keys itself cause any signal? Or does Claude Code's internal message handling kill the running tool?
- Review Claude Code's known behavior: When new user input arrives at the composer while a Bash tool is executing, Claude Code interrupts the current tool execution (this is visible in the TUI as the "interrupt" behavior when you type while a tool runs)

**3. Agent-deck's role (can we prevent it):**
- Examine agent-deck's send path: `sendWithRetry` calls `SendKeysAndEnter` which does `tmux send-keys -l "text"` then `tmux send-keys Enter`. This is equivalent to a human typing in the tmux pane.
- Determine: Is there any difference between agent-deck sending via tmux and a human typing directly? If not, the SIGKILL comes from Claude Code's own interrupt-on-input behavior.
- Check if there's a way to queue messages for Claude Code without interrupting the running tool (e.g., waiting until the tool finishes before injecting text)

**4. Fixability determination:**
- If the SIGKILL comes from Claude Code's internal handling of new input during tool execution: NOT fixable in agent-deck (Claude Code design choice)
- If the SIGKILL comes from tmux's PTY handling or signal propagation: potentially fixable
- If agent-deck could mitigate by timing sends differently (only send when status is "waiting", never when "running"): document this as mitigation
