Read the investigation findings from 09-INVESTIGATION.md and add a new section to ~/.agent-deck/conductor/CLAUDE.md documenting the exit 137 behavior and mitigation.

Add a section titled "## Exit 137: Tool Interruption on Incoming Messages" (or similar based on investigation findings) that covers:

1. **What happens**: When a message arrives (via tmux send-keys, which is what `session send` uses) while Claude Code is executing a Bash tool, Claude Code interrupts the running tool with SIGKILL (exit code 137). This is Claude Code's internal behavior, not caused by agent-deck or tmux.

2. **Why it happens**: Claude Code treats new PTY input during tool execution as an interrupt signal. The send-keys mechanism is indistinguishable from a human typing in the terminal.

3. **Practical mitigations** (derive from investigation findings, expected to include):
   - Check session status before sending: only send when status is `waiting` (agent is idle), not `running` (tool executing). The `session send` command already does this by default (waitForAgentReady), but `--no-wait` bypasses it.
   - Keep CLI commands in child sessions short to minimize the window where they can be interrupted.
   - Use event-driven monitoring instead of sleep+poll loops (events don't require sending messages to running sessions).
   - For heartbeat/status checks, use `--no-wait` only when the session is known to be in `waiting` state.

4. **What does NOT help**: `nohup`, background processes, or signal trapping in the child process (Claude Code sends SIGKILL which cannot be caught).

Place this section after "## Heartbeat Protocol" and before "## State Management" in the existing CLAUDE.md structure. Keep the section concise (aim for 20-30 lines), practical, and focused on what conductors should DO differently.

The GSD conductor skill is used when orchestrating sessions running GSD commands (/gsd:plan-phase, /gsd:execute-phase, etc.). Add a section or subsection covering:

1. **GSD-specific impact**: GSD execute-phase sessions run long Bash tool calls (builds, tests, linting). If the conductor sends a message to a GSD session while it's executing a task, the running tool gets killed with exit 137.

2. **GSD-specific mitigations**:
   - Always check session status before sending. GSD sessions in `running` status are actively executing plan tasks. Wait for `waiting` status.
   - Use `session send --wait` for GSD sessions, which naturally waits for readiness before sending and waits for completion after.
   - For GSD execute-phase sessions, prefer event-driven monitoring (wait for completion events) over periodic polling.
   - If a GSD session shows `running` for an extended period, do NOT send a "status?" message. Check via `session output` (read-only, does not interrupt) instead.

3. **Recovery**: If a GSD session's tool was killed by exit 137, the session will likely show an error and retry or ask for guidance. The session itself is not killed, only the child process of the current Bash tool invocation.

Place this in a logical location in the SKILL.md (likely near the "Critical: Interactive Prompt Handling" section or as a new section after the GSD Lifecycle section). Keep it concise and actionable.
